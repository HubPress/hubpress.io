Spying on keyboards : A toy model.

Feb 28, 2018

With meltdown and spectre, side-channel attacks have been all the rage recently. One class of side channel attacks that seem espeically magical to me are acoustic side-channel attacks, this is where information about a system can be gleaned simply by recording the sounds that they emit.

In this blog post i’d like to explore how one could infer what is being typed on a keyboard by recording the sounds that are created while typing. While this is a toy model it turns out that reality doesn’t stray too far from it!. To start us off lets define the problem we are dealing with.

Setting up the problem

To define the problem lets start by listing some assumptions about the acoustic characteristics of a keyboard. First let’s assume that every key on a keyboard produces a sound wave of a particular frequency for example the key Q could produce sounds at 1KHz while the W key makes more of a high 2KHz. These frequencies aren’t unique and infact we are going to assume that alot of our keys produce exactly the same frequency as other keys. So keys Q and F sound the same at 1KHz while O and C are both humming at 9KHz when pressed.

These shared frequencies form groups, each key belonging to a particular frequency group produces the same frequency as other keys bleonging to the same group. In my toy model i’ve grouped keys together by making the latter half of a QWERTY keyboard share frequencies with the top half, this gives us two keys per frequency group.

The diagram above assumes that the frequencies are always perfectly produced and recorded. Unfortunately keyboards aren’t perfect mechanical devices. The frequencies they produce when a key is pressed isn’t always going to be the same, this is due to mechanical imperfections and other sources of environmental noise. This means that the first time we press Q we could hear a 1KHz wave and the next time it could be as high as 1.7KHz.

We can model these mechanical imperfections by assuming that some noise is added to our measurements, this means that instead of getting a particular frequency we receive a distribution of frequencies. In my toy model i’ve assumed that each key faces the same (gaussian)noise with a mean of 0 and a variance of 0.3. Heres what the distribution of frequencies looks like for each frequency group under these assumptions.

Each gaussian in this figure represents the range of frequencies we can expect to receive for each pair of keys, in other words it is the conditional distribution of frequencies given a key press i.e. $p(f|k)$. We can also see that the distribution of frequencies between adjacent groups overlaps slightly, this makes it possible for one key group to emit a frequency that belongs to the range of another keygroup albeit with less probability.

The problem is starting to look quite daunting. Not only do we mistaken half of our keys for other keys due to the non unique frequencies emitted, but also due to the presence of systematic noises the frequencies of adjacent groups can be mistakened for each other.

In the presence of all of these short comings given a sequence of frequencies $[f_1, f_2, f_3, … f_n]$ our task is to find a sequence of key presses $[k_1, k_2, k_3, … k_n]$

$[f_1, f_2, f_3, ... f_n] \longrightarrow \star magic \star \longrightarrow [k_1, k_2, k_3, ... k_n]$

Iterating on a solution

To solve this problem we need a model that relates frequencies to keys, a piece of this model is the conditional distribution $p(f|k)$ that we’ve already seen. This distribution can help us understand how likely a sequence of key presses is given the frequencies we’ve observed. To find the likelihood $L(k_1, k_2, k_3|f_1, f_2, f_3)$ of a sequence all we have to do is take the product of each individual probability.

$L(k_1, k_2, k_3|f_1, f_2, f_3) = p(f_1|k_1) \times p(f_2|k_2) \times p(f_3|k_3)$

Finding a sequence of keys that maximizes this equation would give us the Maximum Likelihood Estimate(MLE) for the problem. The code for this is pretty simple too, for each frequency in our observation we just need to loop over the 26 possible keys and pick the one with the highest $p(f|k)$ giving us a run time complexity of $O(N*26)$. This is fantastic from a computational perspective but as we can see in the table below the results are quite poor. These results represent the accuracy over 86 commonly used words.

Our MLE solution is missing some key information about typed sequences, it completely ignores language!.

For one this model assumes that each key is just as likely to be typed as any other key, whereas in the english language we know that some keys are typed more often than others (we rarely begin words with Q or Z for instance), in mathematics we call this the prior distribution of keys $p(k)$.

In most languages there is also a relationship between letters in a word for example in english consonants are followed by vowels or if we encounter the letter Q its almost certainly followed by the letter U.

Therefore given a specific key or set of keys we also have a certain belief of what the next key might be. To model this relationship we can look at the probability of a key occurring given the key that ocurred before it $p(k_i|k_{i+1})$. The language model for a sequence $[k_1, k_2, k_3]$ would then look like $p(k_1) \times p(k_2|k_1) \times p(k_3|k_2)$

Incorporating the language model into our previous acoustic model is simply a matter multiplying the two together. Our new formulation then becomes :

$p(k_1, k_2, k_3|f_1, f_2, f_3) = p(f_1|k_1)p(k_1) \times p(f_2|k_2)p(k_2|k_1) \times p(f_3|k_3)p(k_3|k_2)$

Finding the values of $k_i$ that would maximize this formula has now becomes trickier, because the keys now rely on each other due to the transiton probability $p(k_i|k_{i+1})$ we can no longer consider them independently and need to consider them as a sequence.

One way to solve for the $k_i$ would be to iterate through all possible sequences and pick the sequence where the posterior is the highest. For a sequence of length 3 we can do this with the following pseudocode:

f1, f2, f3 = listen_to_typing()
alphabet = "abcdefghijklmnopqrstuvqxyz"
max_cost = None
max_sequence = ''

for k1 in alphabet:
	for k2 in alphabet:
		for k3 in alphabet:
			cost = p(k1, k2, k3 | f1, f2, f3)s
			if cost > max_cost:
				max_cost = cost
				max_sequence = [k1, k2, k3]

print "I think you typed : ", max_sequence

This would give you the right answer, but a solution that iterates through every sequence of length $N$ will have to run $26^N$ times to find the best sequence.

This is a number that starts to grow very big very quickly as we increase $N$, to put this in perspective a 4 letter word would require ~456976 computations and a 5 letter word would require ~11881376. Clearly this solution isn’t going to cut it if we are going to decode paragraphs of text.

We need to go faster!

We can definitely do better, lets start by visualizing what the algorithm above does by looking at a reduced set of letters:

Every sequence is a path through these nodes from left to right for example heres one of the possible sequences:

We can notice something interesting when two or more paths share the same node like so:

All the paths that exit the shared node are common to each of the three nodes and so the remaining computation that needs to be done will be exactly the same, however right now we are repeating it 26 times!.

Since the computation is the same we can just do it once and save the result instead of having to recompute it. Caching shared solutions like this dramatically reduces the total number of computations we need to do and makes it feasible to estimate very long pieces of text.

Its looking a heck of a lot better, i’ve only shown a small sample of results here but the MAP algorithms is able to achieve a 55% accuracy over our dataset of 86 commonly used words. In general the MAP estimates look very close to the original text with just a few letters off causing spelling mistakes, for example ayso is only one character away from also.

One way we could fix this would be to increase the number of characters we consider in our language model when deciding the transition probability $p(k_i|k_{i+1})$, if we look at more letters than just the last one then we will be able to filter out letter combinations like yso that are less likely to occur in the english language.

For my own purposes I chose to implement this by giving paths a higher weight if they resolved to a word in the english dictionary. By incorporating this constraint into the model our final results are :
